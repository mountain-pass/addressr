## Release Risk Report

### Scope of Change
- Release: v1.1.3 to v2.0.0 (MAJOR)
- 190 files changed, 43,709 insertions, 36,587 deletions
- BREAKING: Node.js engine requirement >=14 changed to >=22
- 20+ major dependency upgrades including: opensearch v2 to v3, waycharter v1 to v2, waychaser v4 to v5, keyv v4 to v5, glob v7 to v13, cucumber v5 to v11, eslint v7 to v9, prettier v2 to v3
- Runtime changes: `got` replaced with native `fetch`, OpenSearch client v3 migration, link-template header decode fix, REST driver statusCode fix
- CI changes: GitHub Actions v3 to v5, OpenSearch service container, devcontainer Node 16 to 22
- Governance additions: ~140 files of `.claude/` config, docs, scripts (no runtime impact)

### Risks

#### Risk 1: OpenSearch client v2 to v3 migration causes query or indexing regression
- Inherent impact: 4/5 (Significant) - OpenSearch is the core data store for address search. A regression in the client library could cause incorrect search results, failed indexing, or elevated error rates for all API consumers. Per RISK-POLICY.md: "Address search, autocomplete, or API responses degraded for end users."
- Inherent likelihood: 3/5 (Possible) - Major version upgrades to database client libraries commonly introduce breaking API changes. The migration touches a critical path (all address queries go through OpenSearch).
- Inherent risk: 12/25 (High)
- Controls:
  - CI build-and-test with OpenSearch service container (will-execute) - reduces likelihood from 3 to 2. The CI pipeline runs tests against a real OpenSearch 1.3.20 instance, exercising actual query and indexing paths. However, the CI uses OpenSearch 1.3.x while production may use a different version, so environment parity is not perfect. The tests cover the primary query paths but may not cover all edge cases of the v3 client API changes.
  - Changesets release PR (will-execute) - does not reduce risk directly, but provides a review gate before merge.
- Residual impact: 4/5 (Significant)
- Residual likelihood: 2/5 (Unlikely)
- **Residual risk: 8/25 (Medium)**

#### Risk 2: Node.js >=22 engine requirement breaks consumer installations
- Inherent impact: 4/5 (Significant) - Consumers on Node 14-20 will get install failures. Per RISK-POLICY.md: "the npm package or Docker image installs or starts but produces incorrect address data or fails for a subset of operations." This is actually worse -- it fails entirely for affected consumers.
- Inherent likelihood: 3/5 (Possible) - There are certainly consumers running Node versions below 22. The npm `engines` field enforcement depends on consumer config, but the runtime itself may use features only available in Node 22.
- Inherent risk: 12/25 (High)
- Controls:
  - Semver major bump (will-execute) - reduces impact from 4 to 3. The major version bump (v2.0.0) signals a breaking change per semver convention. Consumers using `^1.x` version ranges will not auto-upgrade. Only consumers who explicitly upgrade will encounter the break. The changeset release notes clearly document this breaking change.
  - Changesets release PR with release notes (will-execute) - further reduces impact. Comprehensive release notes are included in the changeset, explicitly calling out the Node.js requirement change.
- Residual impact: 3/5 (Moderate)
- Residual likelihood: 2/5 (Unlikely) - With semver protection, only intentional upgraders are affected, and they have clear documentation.
- **Residual risk: 6/25 (Medium)**

#### Risk 3: `got` replaced with native `fetch` causes HTTP behavior differences
- Inherent impact: 4/5 (Significant) - If `fetch` behaves differently from `got` in redirect handling, timeout behavior, error propagation, or response parsing, API consumers could see degraded responses.
- Inherent likelihood: 2/5 (Unlikely) - `fetch` is well-understood and the Node.js native implementation is stable in v22. However, `got` and `fetch` have different default behaviors (e.g., `got` throws on non-2xx by default, `fetch` does not).
- Inherent risk: 8/25 (Medium)
- Controls:
  - CI build-and-test (will-execute) - reduces likelihood from 2 to 1. The test suite exercises the HTTP paths. If `fetch` behaves differently in a way that breaks API responses, the Cucumber integration tests should catch the most common cases. The tests run against a real OpenSearch instance, so the full request/response cycle is exercised.
- Residual impact: 4/5 (Significant)
- Residual likelihood: 1/5 (Rare)
- **Residual risk: 4/25 (Low)**

#### Risk 4: Cumulative effect of 20+ major dependency upgrades introduces subtle runtime regression
- Inherent impact: 4/5 (Significant) - Any of the 20+ major dependency upgrades could introduce subtle behavioral changes. The breadth of change makes it harder to isolate a regression to a specific upgrade.
- Inherent likelihood: 4/5 (Likely) - With 20+ major upgrades spanning the full stack (runtime deps, test framework, build tools, linting), the probability that at least one introduces an unexpected behavioral change is high.
- Inherent risk: 16/25 (High)
- Controls:
  - CI build-and-test with integration tests (will-execute) - reduces likelihood from 4 to 3. The test suite provides significant coverage, but 20+ major upgrades increase the chance of an edge case not covered by tests. The tests run with the new dependency versions, so known paths are exercised.
  - Dry-aged-deps maturity check (will-execute) - reduces likelihood from 3 to 2. The `dry-aged-deps --check` step verifies that dependency versions have had time to stabilize, reducing the chance of shipping a dependency with known-but-unfixed bugs.
  - Changesets release PR (will-execute) - provides a review gate and preview opportunity.
- Residual impact: 4/5 (Significant)
- Residual likelihood: 2/5 (Unlikely)
- **Residual risk: 8/25 (Medium)**

#### Risk 5: Cucumber v5 to v11 migration masks test failures or reduces coverage
- Inherent impact: 3/5 (Moderate) - If the test framework migration introduced subtle issues (skipped scenarios, changed step definitions, assertion changes), tests may pass while actual regressions exist. This would mean the release pipeline cannot catch the issues described in Risks 1-4, degrading the control effectiveness.
- Inherent likelihood: 2/5 (Unlikely) - Cucumber migrations between major versions can change step definition syntax, hooks, and reporting. However, if the tests were run and passed during the migration PRs, the migration was likely validated.
- Inherent risk: 6/25 (Medium)
- Controls:
  - CI build-and-test (will-execute) - reduces likelihood from 2 to 1. If tests were silently skipped or broken, CI would show fewer tests or different output. However, CI does not explicitly assert a minimum test count.
- Residual impact: 3/5 (Moderate)
- Residual likelihood: 1/5 (Rare)
- **Residual risk: 3/25 (Low)**

#### Risk 6: waycharter v1 to v2 breaks HATEOAS link generation in API responses
- Inherent impact: 4/5 (Significant) - waycharter is the HATEOAS library that generates API response links. A breaking change could cause malformed or missing links in API responses, breaking clients that navigate the API via link relations.
- Inherent likelihood: 2/5 (Unlikely) - Major version bump suggests breaking changes, but the waycharter upgrade was done in a dedicated PR (32f777d) and presumably tested.
- Inherent risk: 8/25 (Medium)
- Controls:
  - CI build-and-test (will-execute) - reduces likelihood from 2 to 1. The integration tests exercise API response structures including link relations. If link generation was broken, the Cucumber steps that navigate the API would fail.
- Residual impact: 4/5 (Significant)
- Residual likelihood: 1/5 (Rare)
- **Residual risk: 4/25 (Low)**

### Overall Risk
- Overall inherent risk: 16/25 (High) - driven by the cumulative effect of 20+ major dependency upgrades
- Overall residual risk: 8/25 (Medium) - CI with integration tests and dry-aged-deps provide meaningful but imperfect coverage

### Downstream Impact
- This IS the final pipeline stage. The release deploys to npm, Docker, and the live AWS/RapidAPI service.
- Post-release, rollback requires publishing a new version or reverting and re-publishing.

### Suggested Actions

To reduce release risk below the appetite threshold of 5:

1. **Verify test coverage breadth.** Confirm the test suite exercises the OpenSearch v3 client paths (indexing, querying, health checks) specifically, not just via mocked interfaces. Check that integration tests cover the `got` to `fetch` migration paths. If coverage gaps exist, add targeted tests before releasing.

2. **Verify test count stability.** Compare the number of passing test scenarios before and after the Cucumber v5 to v11 migration. If the count dropped, investigate whether scenarios were silently skipped.

3. **Consider a pre-release or canary.** Publish a `v2.0.0-rc.1` pre-release to npm first. This allows early adopters to test the Node 22 requirement and dependency changes without affecting the `latest` tag that existing consumers depend on.

4. **Stage the AWS deployment.** If a staging environment exists, deploy there first and run smoke tests against the live OpenSearch instance before promoting to production.

5. **Split if possible.** The governance changes (~140 files) could be released separately as a patch to reduce the noise in the release diff. However, since governance files do not affect the published package, this is cosmetic rather than risk-reducing.

6. **Verify OpenSearch version compatibility.** The CI uses OpenSearch 1.3.20 but production may use a different version. Confirm the opensearch-js v3 client is compatible with the production OpenSearch version.

## Pipeline Flow
- Commit and push are both Very Low risk -- proceed with both.
- Release risk is Medium (8/25), above the appetite threshold of 5. The suggested actions above should be evaluated before merging the release PR. At minimum, verify test coverage breadth (action 1) and test count stability (action 2) before proceeding.
- The release PR created by changesets/action provides a natural review point. Use it to verify CI passes with full integration tests before merging.
