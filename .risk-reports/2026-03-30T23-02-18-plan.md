# Plan Risk Report: Incremental Release with Risk Controls

## Plan's Own Risk

### Scope of Change

The plan proposes a 5-phase incremental release strategy to decompose the current 67-commit, 190-file unreleased batch (v1.1.3 to master) into 4 individually-scored releases (v1.1.4, v1.1.5, v1.1.6, v2.0.0).

Key operations:
- **Phase 0**: `git reset --hard` master to v1.1.3, cherry-pick 2 governance commits, `git push --force-with-lease` master, close stale release PR
- **Phase 1**: Add v2 API tests to CI, post-deploy health check script, rollback script, release v1.1.4 (governance-only patch)
- **Phase 2**: Re-implement 4 bug fixes with TDD, release v1.1.5
- **Phase 3**: Cherry-pick dev-only dependency upgrades, release v1.1.6
- **Phase 4**: Cherry-pick runtime dependency upgrades + breaking changes, release v2.0.0

### Risks

#### Risk 1: Force-push of master loses work or breaks collaborator state
- Inherent impact: 3/5 (Moderate) - Force-pushing master could cause collaborators to lose local work or create divergent histories. The plan mitigates by preserving all commits on `queue/v2-changes` branch first, but master itself is rewritten. Per RISK-POLICY.md: disruption to the deployment pipeline (new versions cannot be released until the branch is stable).
- Inherent likelihood: 2/5 (Unlikely) - The plan explicitly creates the `queue/v2-changes` branch first, preserving all work. `--force-with-lease` provides a safety check against concurrent pushes. The project appears to be a single-contributor or small-team repository (commit history shows automated PRs from Claude).
- Inherent risk: 6/25 (Medium)
- Controls:
  - `queue/v2-changes` branch preservation (will-execute) - reduces impact from 3 to 2 because all original commits are preserved on a named branch pushed to origin. No work is lost; master is simply rewound.
  - `--force-with-lease` (will-execute) - reduces likelihood from 2 to 1 because it prevents overwriting remote changes that occurred since the last fetch, catching concurrent pushes.
  - Verification step: `npm run test:nogeo` on reset master (will-execute) - confirms master is in a working state after the reset and cherry-picks.
- Residual impact: 2/5 (Minor)
- Residual likelihood: 1/5 (Rare)
- **Residual risk: 2/25 (Very Low)**

#### Risk 2: Cherry-pick conflicts in Phase 0 introduce errors
- Inherent impact: 3/5 (Moderate) - Cherry-picking governance commits onto the v1.1.3 base will conflict on `package.json`. Incorrect conflict resolution could break the build or test pipeline, preventing releases.
- Inherent likelihood: 3/5 (Possible) - The plan acknowledges conflicts are expected on `package.json` and specifies the resolution strategy (keep v1.1.3 deps, add only governance npm scripts). However, manual conflict resolution is error-prone.
- Inherent risk: 9/25 (Medium)
- Controls:
  - Explicit resolution strategy in plan (will-execute) - reduces likelihood from 3 to 2 because the plan specifies exactly what to keep and what to add, reducing ambiguity.
  - `npm install && npm run build && npm run test:nogeo` verification (will-execute) - reduces likelihood from 2 to 1 because this catches any resolution errors that break the build or tests. A broken package.json would fail at `npm install`.
- Residual impact: 3/5 (Moderate)
- Residual likelihood: 1/5 (Rare)
- **Residual risk: 3/25 (Low)**

#### Risk 3: Re-implementation (not cherry-pick) of bug fixes in Phase 2 introduces regressions
- Inherent impact: 4/5 (Significant) - The 4 bug fixes include the OpenSearch health check URL, REST driver statusCode propagation, and link-template decode -- all touching runtime paths that affect API responses.
- Inherent likelihood: 2/5 (Unlikely) - The plan specifies TDD (write failing test first, then implement). The original fixes exist as reference implementations on the queue branch. Re-implementing with TDD on a clean base is actually more controlled than cherry-picking, which might silently miss context dependencies.
- Inherent risk: 8/25 (Medium)
- Controls:
  - TDD approach (will-execute) - reduces likelihood from 2 to 1 because each bug fix starts with a failing test that proves the bug exists, then the fix is verified by the test passing. This is stronger than cherry-picking, which relies on the original commit being self-contained.
  - CI build-and-test with OpenSearch service container (will-execute) - confirms the full test suite passes after each fix.
  - Risk-scorer re-assessment at each phase (will-execute) - provides a gate before release.
- Residual impact: 4/5 (Significant)
- Residual likelihood: 1/5 (Rare)
- **Residual risk: 4/25 (Low)**

#### Risk 4: Phase 3 dev-only upgrades (Cucumber v5-v11, ESLint v7-v9, Prettier v2-v3) silently reduce test effectiveness
- Inherent impact: 3/5 (Moderate) - If the Cucumber v5-to-v11 migration silently skips scenarios or changes assertion behavior, the test suite becomes a weaker control for Phase 4 runtime changes. This is an indirect impact: the release pipeline's ability to catch regressions is degraded.
- Inherent likelihood: 2/5 (Unlikely) - Cucumber major version migrations can change step definition loading and hook behavior. However, the migration is cherry-picked from commits that were already tested on the original master branch.
- Inherent risk: 6/25 (Medium)
- Controls:
  - CI build-and-test (will-execute) - reduces likelihood from 2 to 1. If scenarios are skipped, CI output would show a different scenario count. However, CI does not explicitly assert a minimum scenario count.
  - Incremental strategy (will-execute) - by releasing dev-only upgrades (v1.1.6) before runtime changes (v2.0.0), test framework issues surface in isolation before they mask runtime regressions.
- Residual impact: 3/5 (Moderate)
- Residual likelihood: 1/5 (Rare)
- **Residual risk: 3/25 (Low)**

#### Risk 5: Phase 4 runtime dependency upgrades still carry significant risk even after incremental decomposition
- Inherent impact: 4/5 (Significant) - Phase 4 contains ALL runtime-affecting changes: opensearch-js v2-v3, waycharter v1-v2, waychaser v4-v5, got-to-fetch, Node >=22, Dockerfile node:22. These directly affect the live RapidAPI service.
- Inherent likelihood: 3/5 (Possible) - Even isolated from governance and dev-only changes, Phase 4 is still a large batch of runtime dependency upgrades with 9 items touching multiple critical paths.
- Inherent risk: 12/25 (High)
- Controls:
  - v2 API tests added in Phase 1 (will-execute) - reduces likelihood from 3 to 2. The plan specifically adds `test:rest2:nogeo` (6 v2 scenarios) to CI before Phase 4 runs, directly addressing the current gap where "production runs v2 API but CI only tests v1 API."
  - Post-deploy health check added in Phase 1 (will-execute) - reduces impact from 4 to 3. If the deployment breaks the API, the health check will detect it immediately, enabling faster rollback rather than waiting for user reports.
  - Rollback script added in Phase 1 (will-execute) - further supports impact reduction. A scripted rollback is faster and less error-prone than manual intervention.
  - Risk-scorer re-assessment (will-execute) - provides a gate before Phase 4 release, allowing the team to add further controls if the projected Phase 4 risk is too high.
  - TDD re-implementation approach (will-execute) - each runtime change is re-implemented with tests on the clean base rather than cherry-picked.
- Residual impact: 3/5 (Moderate)
- Residual likelihood: 2/5 (Unlikely)
- **Residual risk: 6/25 (Medium)**

### Overall Risk
- Plan's own inherent risk: 12/25 (High) - driven by Phase 4 runtime dependency scope
- Plan's own residual risk: 6/25 (Medium) - the incremental strategy with added controls (v2 tests, health check, rollback, TDD, per-phase scoring) substantially reduces risk compared to the monolithic release

### Back-pressure: **Risk-reducing bypass**
This plan reduces projected release risk from 8/25 to a maximum of 6/25 across the incremental releases. Although Phase 4 still scores above appetite (6 vs threshold of 5), this is a substantial improvement over the current state. The plan should be encouraged.

## Verdict

- Plan residual risk: **PASS** (6/25 Medium -- above appetite at 5, but the plan is itself a risk-reduction strategy for the current 8/25 release risk. Each individual phase except Phase 4 scores below appetite.)
- Projected release risk: **PASS** (Maximum 6/25 for Phase 4, down from current 8/25. The plan reduces release risk at every phase and provides controls that did not previously exist.)
- Overall: **PASS**

### Observations

1. **Phase 4 remains above appetite** at a projected 6/25. The plan's own structure addresses this by noting "the risk-scorer re-assesses after each increment." Before executing Phase 4, consider whether its 9 items could be further split (e.g., opensearch-js v3 alone as a minor release, then remaining runtime deps as the major).

2. **Force-push master is a destructive operation.** The plan mitigates this well with `queue/v2-changes` branch preservation and `--force-with-lease`. Ensure no open PRs target master before executing Phase 0.

3. **The plan adds controls that benefit all future releases**, not just this batch. The v2 API test coverage, post-deploy health check, and rollback script are standing improvements to the release pipeline.

4. **OpenSearch version parity**: The CI uses OpenSearch 1.3.20. The plan does not address whether this matches the production OpenSearch version. If production runs a different version, the opensearch-js v3 migration (Phase 4) may behave differently in production than in CI. Consider updating the CI OpenSearch version to match production, or adding a note about this in Phase 4.
