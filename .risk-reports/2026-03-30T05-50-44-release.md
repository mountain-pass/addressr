## Release Risk Report

### Scope of Change
- Major version release: v1.1.3 to v2.0.0
- Breaking change: Node.js engine requirement >=14 to >=22
- 20+ major version dependency upgrades across both runtime and dev dependencies
- Runtime dependency changes: opensearch v2 to v3, waycharter v1 to v2, dotenv v10 to v17, glob v7 to v13, keyv v4 to v5, js-yaml v3 to v4, json-ptr v2 to v3, wait-port v0.2 to v1
- Removed packages: ngrok, imagemin-lint-staged, babel-preset-env, glob-promise
- Replaced got with native fetch
- Bug fixes: OpenSearch health check URL, statusCode propagation, link-template header decoding
- Source code changes mostly prettier reformatting (semicolons, trailing commas)
- CI infrastructure overhaul: GitHub Actions v3 to v5, OpenSearch service container

### Risks

#### Risk 1: Runtime dependency major version incompatibilities
- Inherent impact: 4/5 (Significant) - opensearch v2 to v3, keyv v4 to v5, glob v7 to v13, and js-yaml v3 to v4 all have breaking API changes. If any call site was not updated correctly, address search/autocomplete could fail or return incorrect results for RapidAPI consumers. The opensearch client upgrade directly affects the core data store.
- Inherent likelihood: 3/5 (Possible) - Multiple major version jumps across core runtime dependencies. While prettier reformatting suggests systematic changes were made, the sheer number of major upgrades creates reasonable chance that an edge case was missed. The opensearch v2 to v3 change is particularly sensitive as it touches the primary data store.
- Inherent risk: 12/25 (High)
- Controls:
  - CI build-and-test with OpenSearch service (will-execute) - runs test suite against a real OpenSearch 1.3.20 instance. This directly exercises the opensearch client upgrade. If the v3 client has incompatible API calls, tests that interact with OpenSearch would fail. Reduces likelihood for opensearch-specific failures.
  - CI test suite `npm run test:nogeo` (will-execute) - exercises address search and API endpoint behaviour. If glob, js-yaml, or keyv changes broke loading or caching logic, tests covering those paths would fail. However, test coverage of edge cases for all 20+ upgraded libraries is uncertain.
  - Changeset release PR (will-execute) - creates a PR that can be reviewed before merge. Human review opportunity.
- Residual impact: 4/5 (Significant)
- Residual likelihood: 2/5 (Unlikely) - CI with real OpenSearch reduces likelihood for the most critical dependency (opensearch client). Test suite provides coverage for primary code paths. However, not all edge cases across all 20+ upgrades can be guaranteed covered.
- **Residual risk: 8/25 (Medium)**

#### Risk 2: Node.js >=22 breaking existing consumer installations
- Inherent impact: 4/5 (Significant) - npm package consumers on Node 14-20 will get a post-install failure when upgrading to v2.0.0. This is a deliberate breaking change, but consumers who auto-upgrade without reading release notes will experience breakage.
- Inherent likelihood: 3/5 (Possible) - Some consumers may have automated dependency update tooling (Dependabot, Renovate) that creates PRs for major bumps. The engine field will cause npm install failures, but the failure mode is clear and early (install time, not runtime). However, the jump from >=14 to >=22 skips 4 major Node versions.
- Inherent risk: 12/25 (High)
- Controls:
  - Major version bump declared (will-execute) - semver signals breaking change. Package managers will not auto-upgrade across major versions without explicit configuration.
  - Detailed release notes in changeset (will-execute) - clearly documents the Node.js requirement change as the first breaking change. Consumers who read release notes will be warned.
  - npm engines field (will-execute) - npm will fail fast at install time with a clear error, not at runtime with a cryptic failure. This converts runtime risk to install-time risk.
- Residual impact: 3/5 (Moderate) - With semver major bump and engines field, the failure is at install time (not runtime) and clearly signalled. Impact downgraded because existing installations continue working on v1.1.3. Only consumers who explicitly upgrade are affected, and they get a clear failure.
- Residual likelihood: 2/5 (Unlikely) - Semver major bump prevents accidental auto-upgrades. Engines field ensures fast failure. Release notes document the change.
- **Residual risk: 6/25 (Medium)**

#### Risk 3: got-to-fetch replacement breaking HTTP behaviour
- Inherent impact: 4/5 (Significant) - If the fetch replacement has different error handling, redirect behaviour, timeout behaviour, or response parsing compared to got, API responses to RapidAPI consumers could be affected.
- Inherent likelihood: 2/5 (Unlikely) - fetch is a well-understood API. The replacement was likely tested during development. CI test suite exercises HTTP endpoints.
- Inherent risk: 8/25 (Medium)
- Controls:
  - CI build-and-test (will-execute) - test suite exercises HTTP API behaviour. If fetch handles responses differently from got in ways that affect test assertions, tests would fail. This directly covers the HTTP layer.
- Residual impact: 4/5 (Significant)
- Residual likelihood: 1/5 (Rare) - CI tests directly exercise HTTP endpoints, reducing likelihood that a fetch behavioural difference would go undetected.
- **Residual risk: 4/25 (Low)**

#### Risk 4: Prettier reformatting introducing subtle code changes
- Inherent impact: 4/5 (Significant) - If prettier v2 to v3 reformatting introduced any semantic changes (not just whitespace/formatting), it could cause runtime failures in address processing.
- Inherent likelihood: 1/5 (Rare) - Prettier is a formatter, not a transpiler. Its changes are syntactic (semicolons, trailing commas, quote style). Semantic code changes from prettier are extremely rare. The CI test suite would catch any accidental semantic changes.
- Inherent risk: 4/25 (Low)
- Controls:
  - CI build-and-test (will-execute) - if any semantic change was introduced, tests would catch it
- Residual impact: 4/5 (Significant)
- Residual likelihood: 1/5 (Rare)
- **Residual risk: 4/25 (Low)**

### Overall Risk
- Overall inherent risk: 12/25 (High) [from Risk 1 and Risk 2]
- Overall residual risk: 8/25 (Medium) [from Risk 1: runtime dependency incompatibilities]

### Suggested Actions
- Before merging the release PR, verify that the CI test suite specifically exercises OpenSearch v3 client operations (index creation, document indexing, search queries) — not just connectivity
- Consider running a manual smoke test against a local OpenSearch instance with the full G-NAF dataset loaded to validate address search quality is unchanged
- Review the opensearch v2-to-v3 migration guide and confirm all deprecated API calls have been updated
- Verify that `keyv` v5 cache behaviour is tested (cache hit, cache miss, cache expiration)
- Consider whether the release could be split: ship dependency upgrades (without Node >=22 change) as v1.2.0 first, then ship the engine requirement as v2.0.0 — though this may not be practical given the dependency chain
