# Problem 008: RapidAPI gateway returns 401 "Invalid API key" for every key on the Addressr listing

**Status**: Closed
**Reported**: 2026-04-15
**Resolved**: 2026-04-15
**Closed**: 2026-04-16
**Priority**: 25 (Very High) — Impact: Severe (5) x Likelihood: Almost certain (5)

## Description

The RapidAPI gateway has begun rejecting **every** request to the Addressr API listing (api ID `api_5f12dd58-ab01-419e-8c9b-91d19208d16a`) with `HTTP 401 {"message":"Invalid API key. Go to https://docs.rapidapi.com/docs/keys for more info."}`, including:

- Requests from paying consumers using their own valid RapidAPI keys (`Midnight Health-Default`, `Le Reve-Default`, others)
- Requests from the addressr.io homepage example via the Cloudflare Worker key proxy (ADR 018)
- Requests from RapidAPI Studio using the **provider's own owner key**

The 401 response is generated by the RapidAPI gateway (response headers include `server: RapidAPI-0.0.30`, `x-rapidapi-proxy-response: true`); no traffic reaches the upstream Express origin (`backend.addressr.io`).

The Express origin and OpenSearch backend are healthy and unrelated to this incident — direct calls to `https://backend.addressr.io/...` return 200.

## Symptoms

- All RapidAPI consumer requests fail with 401, regardless of key, plan, or endpoint.
- addressr.io homepage "Try it" example fails (homepage widget calls `api.addressr.io` → Cloudflare Worker → RapidAPI gateway → 401).
- UptimeRobot "Addressr Direct" monitor reports DOWN (the monitor is also routed through the Cloudflare Worker → RapidAPI gateway).
- RapidAPI dashboard daily health check shows FAILURE (separate issue — RapidAPI's own prober does not authenticate; pre-existing false positive).
- Provider's own key, tested in RapidAPI Studio against the production gateway, returns 401.
- Backend origin (`backend.addressr.io/health`, `/addresses`, `/api-docs`) returns 200 throughout — the incident is entirely in the RapidAPI auth layer.

## Workaround

**For the API service:** none from our side — the listing's auth layer is on RapidAPI's platform and out of our control until they respond.

**For affected paying consumers (interim):** offer a temporary direct upstream URL `https://backend.addressr.io/...`, which currently has open CORS and no auth requirement. This bypasses RapidAPI entirely and restores service. Caveats:

- Loses RapidAPI's per-customer quota tracking and revenue accounting
- Exposes the upstream that ADR 017 places behind RapidAPI as the gateway
- Should be revoked once RapidAPI is restored
- Already a security concern flagged for follow-up (P009 candidate — make Express origin require `X-RapidAPI-Proxy-Secret`)

## Impact Assessment

- **Who is affected**: All paid and free-tier RapidAPI consumers of the Addressr listing. The addressr.io public website demo. UptimeRobot monitor. Per RISK-POLICY.md Impact level 5 — "loss of revenue-generating capability affecting paid subscribers".
- **Frequency**: Continuous since incident onset (~2026-04-14 evening AEST, based on first observed failed monitor and consumer logs).
- **Severity**: Severe — complete loss of monetised API surface. 17+ subscribers visible on the listing; named paying consumers observed in failure logs.
- **Analytics**: RapidAPI Studio → Requests log shows multiple consumers, all 401/429. RapidAPI dashboard shows listing as Published with healthy structural state.

## Root Cause Analysis

### Confirmed not the cause

- **Backend origin / Express service**: healthy, returning 200 to direct calls.
- **OpenSearch / G-NAF data**: healthy, queries succeed.
- **Cloudflare Worker (`cool-bush-ca66.addressr-key-provider.workers.dev`)**: running, configured correctly, route bound to `api.addressr.io/*`, hardcoded key correctly attached. The 401 it forwards is from RapidAPI gateway, not the worker itself.
- **Cloudflare DNS / hostname routing**: `api.addressr.io` correctly routes to the worker; `backend.addressr.io` correctly routes to origin.
- **Listing structural state**: General, Definitions/Endpoints, Definitions/Security (correctly empty per RapidAPI docs), Gateway, OAS3 all appear healthy in the RapidAPI Studio dashboard.

### Confirmed cause (proximate)

The RapidAPI gateway is rejecting every `X-RapidAPI-Key` validation attempt for this listing — including the provider's own owner key tested from RapidAPI Studio against the production gateway URL `https://addressr.p.rapidapi.com/addresses?q=...`. Because the same rejection happens for the owner's own key, the cause is a platform-side state on RapidAPI's listing, not per-consumer.

### Confirmed underlying cause

A manual OpenAPI spec import into the RapidAPI listing earlier in the day silently mutated the **Hub Listing → Base URL** configuration. The listing had two upstream URLs configured (Round Robin load balancer):

- `https://backend.addressr.io` — correct
- `https://backend2.addressr.io` — **previously had been a `addressr.p.rapidapi.com`-style entry, was overwritten by the OAS3 import**

Both URLs in fact resolve to healthy upstreams (verified during diagnosis), so the load-balancer destinations themselves were not the failure mode. The OAS3 import made other listing-state changes as a side effect — the visible URL change being the most obvious — and one of those side effects caused the gateway to reject all `X-RapidAPI-Key` validation attempts (consumer keys and the provider's own owner key).

The exact field that triggered the auth rejection was not isolated during recovery — once the listing config was restored/re-saved, the gateway resumed accepting keys. RapidAPI's own UI does not surface a per-import diff or audit log, which is itself a contributing factor for diagnosis time.

### Recovery confirmed

- UptimeRobot "Addressr Direct" monitor flipped to UP at 2026-04-15 ~08:00 AEST
- Direct probe of worker URL: `GET https://cool-bush-ca66.addressr-key-provider.workers.dev/addresses/GANSW718804790` → 200 with full HATEOAS address record
- Direct probe of homepage path: `GET https://api.addressr.io/addresses?q=sydney` → 200 with real results

### Lessons (for follow-up)

1. **OAS3 import to RapidAPI is a destructive operation against listing state.** The import does not cleanly map "spec → endpoints"; it can mutate Base URL load-balancer entries and other listing-config fields without warning. Future spec-sync work (P006, ADR 023) must treat import as a high-risk operation and gate it behind a manual review of the diff.
2. **The failure mode masquerades as auth failure.** The gateway emitted 401 "Invalid API key" for what was really a listing-config-state issue. Diagnostic time was lost ruling out the wrong failure modes (worker key, consumer keys, billing).
3. **No automated alerting caught this.** UptimeRobot detected the symptom but the alert was treated as a vendor false positive given prior RapidAPI health-check noise. A direct probe of `addressr.p.rapidapi.com` with the owner key from a synthetic monitor would have shouted earlier.

### Reproduction

Run from any client (or RapidAPI Studio against `addressr.p.rapidapi.com`):

```bash
curl -s -o /dev/null -w "%{http_code}\n" \
  -H "x-rapidapi-key: <ANY_VALID_KEY_INCLUDING_OWNER>" \
  -H "x-rapidapi-host: addressr.p.rapidapi.com" \
  "https://addressr.p.rapidapi.com/addresses?q=sydney"
# Expected when healthy: 200
# Observed (incident): 401 {"message":"Invalid API key..."}
```

When the same call succeeds, the incident is resolved at the RapidAPI layer.
